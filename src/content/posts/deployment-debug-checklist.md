# 从一次失败部署说起：我的排障流程清单

线上事故最怕“凭感觉试错”。  
我现在固定用一个流程，把问题从“模糊焦虑”变成“可验证步骤”。

## 先收集，再下结论

第一步永远不是改代码，而是收集事实：

- 报错发生在什么时间
- 影响范围是全量还是部分请求
- 最近一次上线变更包含哪些模块

没有这些信息时，任何判断都只是猜测。

## 划分故障层级

我会按层级排查，避免跳步：

1. 基础设施层：网络、DNS、证书、网关
2. 运行时层：容器、进程、内存、CPU
3. 应用层：配置、依赖、业务逻辑

每层只回答一个问题：这一层是否能被证伪？

## 最小复现优先

如果无法在本地复现，就在预发布环境构造最小用例。  
最小复现的价值是：你可以快速验证“修复是否真的有效”。

## 记录修复，不只记录结果

我会在故障文档里写三段：

- 触发条件
- 诊断路径
- 预防动作

这样下一次同类问题出现时，团队可以直接复用。

## 清单模板

```txt
[ ] 是否确认影响范围
[ ] 是否定位到具体层级
[ ] 是否完成最小复现
[ ] 是否验证回滚方案
[ ] 是否更新故障文档
```

排障不是英雄主义，稳定流程比灵光一现更可靠。
