# Agent Guide Phase 2: Core Interaction Pages & Admin Layout (miku-blog-ai)

## 🎯 阶段目标与设计延续

本阶段聚焦于博客系统的核心交互页面与后台管理壳层布局，目标是在 **Phase 1 已完成的品牌视觉与页面基调** 之上，补齐用户进入系统、进行互动、查看友链、后台管理内容的完整前端体验闭环。

本阶段必须严格继承 Phase 1 的视觉规范，不允许自行引入新的视觉语言或组件体系。全局主题色固定为：

```css
--miku-color: rgb(102, 205, 170);
```

中文字体首选 **「霞鹜文楷」**，英文字体与数字展示优先使用 **JetBrains Mono / Fira Code / Inter** 等现代字体组合，以维持“优雅 + 技术感”的统一调性。

所有涉及卡片、面板、信息容器、悬浮块的 UI 元素，**必须强制复用**：

* `frontend/src/components/LiquidGlassCard.vue`

以保持液态玻璃（Glassmorphism / Liquid Glass）风格的一致性。严禁同类区域自行改用普通白底卡片、纯色 block、无模糊背景容器等替代实现。

---

## 🧭 本阶段页面范围

本阶段必须完成以下四类页面：

1. 登录页面 `src/pages/login.astro`
2. 幕后管理页 `src/pages/admin/index.astro`
3. 留言板页面 `src/pages/guestbook.astro`
4. 友情链接页面 `src/pages/friends.astro`

这些页面均应基于 **Astro 页面壳 + Vue 3 Islands 交互岛屿** 的架构实现。静态结构交由 Astro 输出，所有需要本地状态、表单输入、切换动画、异步提交、Loading 控制的模块，必须拆分为按需加载的 Vue 组件挂载。

## 一、全局实现约束（更新版，必须严格遵守）

### 1. 技术栈与架构边界

* 前端技术栈固定为 **Astro + Vue 3 Islands + Nano Stores + Tailwind CSS**。
* 包管理器 **只能使用 npm**，禁止切换为 pnpm、yarn、bun。
* 页面必须遵循 **Astro 负责页面壳与静态结构，Vue Islands 负责局部交互** 的架构边界。
* 所有需要响应式状态、表单输入、异步请求、显隐切换、局部动画的模块，必须拆分为按需加载的 Vue 组件。
* 严禁将整页粗暴实现为单个大型 Vue 页面，破坏 Astro 的轻量渲染优势。
* 统一使用 **Vue 3 Composition API**，禁止 Options API 与组合式 API 混写。
* 禁止引入 React、Pinia、Vuex、Element Plus、Ant Design Vue、Naive UI 等额外框架或重量级 UI 库。

### 2. 视觉系统继承约束

* 本阶段必须 **100% 继承 Phase 1 的设计语言**，不得自行创造新的页面风格。
* 全局主题色固定为：

```css
--miku-color: rgb(102, 205, 170);
```

* 中文字体首选 **霞鹜文楷**，英文与数字优先使用 **Inter / JetBrains Mono / Fira Code**。
* 所有具备卡片、面板、浮层、信息区块、输入容器属性的 UI 元素，**必须优先复用** `LiquidGlassCard.vue`。
* 严禁在新页面中出现与既有页面风格冲突的纯白硬边卡片、重阴影卡片、Material 风格组件或传统后台管理系统默认视觉。
* 深色模式与浅色模式都必须保持液态玻璃质感一致，不能一边高级、一边塌成半透明塑料片。

### 3. 组件复用约束

* 凡是重复出现两次及以上的结构，必须优先抽离为公共组件，禁止页面内复制粘贴实现。
* 本阶段建议重点复用或抽离的组件包括但不限于：

  * `LiquidGlassCard.vue`
  * `MikuButton.vue`
  * `MikuInput.vue`
  * `MikuTextarea.vue`
  * `LoadingSpinner.vue`
  * `EmptyState.vue`
  * `ErrorState.vue`
  * `MacWindowToolbar.vue`
  * `AdminSidebar.vue`
  * `DashboardStatCard.vue`
  * `GuestbookMessageCard.vue`
  * `FriendLinkCard.vue`
* 页面文件职责仅限于页面拼装与数据组织，复杂交互逻辑不得长期滞留在 `.astro` 页面内部。
* 公共组件必须具备明确 Props 接口，不允许通过大量魔法 class 或隐式插槽约定让项目逐渐变成黑箱沼泽。

### 4. 样式实现约束

* **严禁脱离 Tailwind 类名体系直接写内联样式**。
* 复杂样式仅在以下场景允许写入 `<style scoped>`：

  1. Tailwind 难以表达的关键帧动画
  2. 特殊遮罩、滤镜、伪元素效果
  3. 与 `LiquidGlassCard` 视觉机制高度绑定的局部实现
* 禁止为了省事大量使用魔法数字、随意的 `style="margin-left: 13px"` 之类的野路子。
* 所有颜色、圆角、阴影、模糊、透明度应尽量通过 Tailwind 语义化组合保持一致。
* 间距、字号、圆角层级必须统一，不允许一个页面圆角像鹅卵石，另一个页面像乐高积木。

### 5. 动效与交互反馈约束

* 动效基调必须保持 **轻、缓、柔**，避免夸张、廉价、炫技式动画。
* Hover、Focus、Active、Loading、Disabled 五种交互状态必须明确可见。
* 统一建议：

  * Hover 过渡：`duration-300` 或 `duration-500`
  * 面板显隐：`opacity + translate-y`
  * 新元素插入：淡入 + 轻微滑入
  * 按钮点击：轻微缩放或阴影变化
* 禁止出现强烈抖动、弹飞、持续闪烁、频繁旋转等破坏气质的动效。
* 所有异步操作都必须给出明确反馈，禁止“用户点了按钮但界面装死”。

### 6. 状态管理约束

* 全局共享状态必须通过 **Nano Stores** 管理，禁止页面内部各自维护一套互不相认的状态逻辑。
* 至少应统一管理以下状态域：

  * `loading`：全局加载态与局部提交态
  * `auth`：登录态、用户信息、Token 存在性
  * `ui`：Sidebar 展开/折叠、全局浮层、Toast 等 UI 状态
  * `guestbook`：留言列表、提交状态、本地插入
  * `friends`：友链列表加载状态与后续筛选能力
* 任何异步流程必须明确区分：

  * `idle`
  * `loading`
  * `success`
  * `error`
* 禁止只用一个布尔值糊住全世界，这种写法通常会在第三个交互场景开始闹鬼。

### 7. 表单与校验约束

* 所有表单类交互必须具备前端基础校验，不能把空值、无效值、异常格式原封不动扔给后端。
* 登录页至少校验：

  * 用户名/邮箱不能为空
  * 密码不能为空
* 留言板至少校验：

  * 昵称不能为空
  * 留言内容不能为空
  * 网址若填写必须格式合法
* 错误反馈必须使用页面内可见提示，不得依赖浏览器原生 `alert()` 作为主反馈方式。
* 提交中必须禁用重复点击。
* 提交失败时必须保留用户已输入内容，禁止失败后直接清空输入框，像把访客留言扔进异次元垃圾桶。

### 8. 路由与权限约束

* `/admin` 及后续后台子路由必须视为受保护页面。
* 无登录态访问后台页时，必须自动跳转到 `/login`。
* 已登录用户访问 `/login` 时，应自动重定向至 `/admin`。
* 鉴权判断应尽量避免页面闪烁，必要时应提供初始化 Loading 壳层。
* 权限控制逻辑必须与 UI 展示逻辑解耦，不能把“是否显示组件”当成完整权限系统。

### 9. 数据请求与错误处理约束

* 当前阶段若后端接口未完全落地，可使用 Mock 数据，但组件设计必须按真实接口接入方式组织。
* 所有数据请求至少要覆盖：

  * Loading 状态
  * 空状态
  * 错误状态
  * 成功状态
* 列表页不得只处理“成功有数据”这一种童话分支。
* 错误提示必须可读、克制、可恢复，必要时提供“重试”入口。
* 禁止静默吞错，控制台红了一片而页面岁月静好，这种前端最会演戏。

### 10. 响应式与布局约束

* 优先保证桌面端视觉完整度，但必须兼容平板与移动端。
* 登录页在小屏幕下必须保持居中与完整可操作，不得出现输入框被键盘或视口挤爆的问题。
* 后台页 Sidebar 在中小屏设备上必须支持折叠、抽屉或覆盖式替代方案。
* 留言板与友链页在手机端必须自然退化为单列布局，禁止横向滚动。
* 所有页面内容宽度必须有阅读边界，不能无节制撑满屏幕，看起来像把设计稿放进了离心机。

### 11. 可访问性约束

* 所有按钮、图标按钮、输入框、密码显隐切换控件必须具备可访问标签，如 `aria-label`。
* 焦点态必须清晰可见，禁止为了“干净”而移除 focus ring。
* 键盘 Tab 顺序必须符合自然阅读流。
* 图标不能作为唯一语义载体，必要时应有文字说明或屏幕阅读器可读标签。
* 文本对比度必须达标，尤其在玻璃背景、半透明层和深色模式下不能牺牲可读性。

### 12. 性能与加载约束

* Vue Islands 必须按需挂载，避免无意义的客户端全量水合。
* 能静态输出的内容优先静态输出，能局部交互的不要整页交互化。
* 图片、头像、背景资源必须考虑体积与加载策略，禁止直接塞入超大原图拖垮首屏。
* 动画与模糊效果应控制在合理范围，避免多层 backdrop-blur 叠加导致低端设备掉帧。
* 后台页与留言板列表页在 mock 或真实数据场景下都要预留骨架屏方案。

### 13. 文件组织约束

* 页面放在 `src/pages`
* 通用组件放在 `src/components`
* 页面专属组件可按功能拆入局部目录，但不得与通用组件混乱堆放
* 全局状态放在 `src/stores`
* 工具函数放在 `src/lib` 或 `src/utils`
* 类型定义建议集中在 `src/types`
* 文件命名保持一致性：

  * 组件：PascalCase
  * Store：camelCase 或语义明确的短名
  * 页面路径：遵循 Astro 文件路由规范

### 14. 禁止事项（硬性红线）

* 禁止引入 Pinia / Vuex 替代 Nano Stores
* 禁止引入 React 或任意与 Vue Islands 冲突的运行时
* 禁止使用 GORM、Gin、Fiber 等后端框架语义污染前端代码
* 禁止使用内联样式作为常规实现方式
* 禁止复制粘贴造多个相似卡片而不抽组件
* 禁止只做“静态摆拍页面”而不考虑真实交互状态
* 禁止新增与 Phase 1 不一致的设计语言
* 禁止为了炫效果牺牲可读性、可维护性与性能

# 二、共享交互逻辑与状态设计

## 1. Nano Stores 全局状态建议

本阶段涉及明显的全局交互反馈，建议新增统一状态 Store，而不是每个页面各自乱管。

建议至少包含：

* `loading.ts`

  * 管理全局 Loading 状态
  * 用于登录提交、留言提交、后台数据加载等场景
* `auth.ts`

  * 保存当前登录态、用户基础信息、Token 是否存在
  * 控制 `/admin` 页面是否展示受保护内容
* `ui.ts`

  * 管理 Sidebar 展开/收起
  * 管理全局 Toast / Modal / 浮层显隐
* `guestbook.ts`

  * 管理留言提交后的本地插入、列表刷新、乐观更新
* `friends.ts`

  * 管理友链列表加载状态、筛选状态（若后续扩展）

## 2. 加载与反馈规范

任何异步操作必须有明确的三态反馈：

* Loading：按钮进入禁用态，显示旋转 SVG / 骨架屏 / 半透明遮罩
* Success：出现轻量级成功反馈（如 Toast、按钮状态恢复、列表插入动画）
* Error：展示明确错误文案，不允许静默失败

禁止“点了按钮没反应”的幽灵交互。这种东西非常像前端里的小鬼，会严重破坏体验。

## 3. 表单校验规范

所有输入表单必须包含基础前端校验：

* 登录页：用户名/邮箱不能为空，密码不能为空
* 留言板：昵称不能为空，留言内容不能为空，网址为可选但若填写必须格式合法
* 友链申请按钮若后续接入弹窗表单，也必须遵循同样原则

错误提示应使用轻量、克制的方式呈现：

* 输入框下方浅红色提示文字
* 或输入框边框变为柔和红色
* 不允许使用粗暴的浏览器原生 alert 作为主要错误反馈

# 三、页面公共视觉语言补充规范

## 1. 动画统一基调

本项目动效必须维持“轻、缓、柔”的气质，避免后台页面做成廉价炫技控制台。

统一建议：

* Hover 动画时长：`duration-300` 或 `duration-500`
* 面板显隐：`ease-out` + 轻微 `translate-y`
* 新元素插入：淡入 + 上浮 / 下滑，不允许硬切
* 按钮点击：轻微 `scale-[0.98]` 或阴影变化，不做夸张弹跳
* Glassmorphism 元素不允许叠加过强阴影，避免脏重感

## 2. 颜色层级

* 主强调色仅允许使用 `--miku-color`
* 背景层、边框层、文本层、辅助文字层要有明确对比
* 灰色信息（时间戳、描述文案、占位提示）必须统一收敛，不可喧宾夺主
* 深色模式下必须保证：

  * 输入框边界清晰
  * 文本可读
  * 主题色不刺眼
  * 毛玻璃不会因透明度失衡而“糊成一锅粥”

## 3. 可访问性

* 所有按钮、输入框、密码切换图标都必须有可访问标签（`aria-label`）
* 键盘 Tab 顺序必须自然
* 焦点态必须清晰可见，不允许为了“好看”把 focus ring 全部抹掉
* 图标按钮 hover 与 focus 态必须一致可辨识

# 四、页面级实现极致细节规格

## 1. 登录页面 (Login View)

路径：`src/pages/login.astro`

### 页面定位

这是用户进入后台系统的入口页，不是普通表单页面，而是带有强烈沉浸感和品牌气质的“系统入口界面”。风格参考 macOS 锁屏页，但必须保留博客项目自身的主题感，而非机械模仿。

### 视觉结构

* 整页高度至少占满一个视口 `min-h-screen`
* 背景为全屏动态壁纸或静态背景图叠加高斯模糊遮罩
* 可在背景上附加轻微暗角、渐变层、透明噪点层，增强空间感
* 屏幕中央垂直水平居中放置一个紧凑的 `LiquidGlassCard.vue`
* 卡片宽度建议控制在 `w-full max-w-md` 范围内，避免过宽失去锁屏精致感

### 卡片内部布局

建议结构顺序为：

1. Logo / 站点名称
2. 圆形 Avatar
3. 欢迎语 / 简短副标题
4. 用户名或邮箱输入框
5. 密码输入框
6. 登录按钮
7. 可选的底部辅助文案（如“仅管理员可访问后台”）

### 头像区

* Avatar 居中展示
* 外圈带极轻的主题色微光
* 鼠标悬浮时可有轻微 scale 与阴影增强
* 不允许使用过重拟物边框

### 输入框组

* 采用极简风格
* 可使用以下两种之一，但全页必须统一：

  * 底边框风格
  * 浅透明背景 + 圆角风格
* 获取焦点时：

  * 边框平滑过渡到 `--miku-color`
  * 可伴随轻微外发光
* 占位文字颜色弱化，避免与真实输入混淆
* 输入框高度、内边距、圆角必须一致

### 密码切换

* 密码框右侧必须内嵌“小眼睛”图标按钮
* 图标使用单色细线 SVG
* 点击后在 `password / text` 间切换
* 图标按钮需有 hover / focus 状态
* 切换行为不能导致输入框高度抖动或布局偏移

### 登录按钮

* 宽度 100%
* 使用高对比主题色按钮
* 文案默认显示“登录” / “Sign In”
* 点击提交后：

  * 按钮进入禁用态
  * 文字替换为加载中动画
  * Nano Stores 更新全局 loading 状态
* 成功后可跳转到 `/admin`
* 失败后卡片内显示温和但明确的错误提示

### 交互动效

* 登录卡片首次进入时：淡入 + 轻微上浮
* 输入框 Focus：边框与阴影过渡
* 按钮 Hover：亮度轻微提升或阴影增强
* 按钮 Active：轻微缩放
* 整页动效必须安静，不允许像赛博酒吧招牌一样乱闪

### 异常态与边界情况

* 已登录用户访问 `/login` 时应自动重定向到 `/admin`
* 登录失败需展示错误文案
* 网络异常需展示通用失败提示
* 提交中禁止重复点击
* 输入为空时直接前端阻止提交

## 2. 幕后管理页 (Admin Dashboard)

路径：`src/pages/admin/index.astro`

### 页面定位

这是后台系统的首页壳层，不仅是一个“数据展示页”，也是未来文章管理、评论审核、友链管理等子模块的布局模板。它必须先把后台的空间秩序建立起来。

### 视觉结构

* 采用经典双栏布局：

  * 左侧固定 Sidebar
  * 右侧 Main Content 自适应撑满
* 整页需具备桌面系统般的秩序感，风格参考 macOS Finder / modern admin console
* 背景使用纯净的浅色或深色大底，突出前景液态玻璃卡片

### 推荐布局拆分

建议拆为：

* `AdminLayout.vue`
* `AdminSidebar.vue`
* `AdminTopbar.vue`
* `DashboardStatsGrid.vue`
* `DashboardQuickPanel.vue`

### 左侧边栏 (Sidebar)

* 固定宽度建议 `w-60` 或 `w-64`
* 使用 `LiquidGlassCard.vue` 或基于其风格包裹的容器
* 导航项至少包含：

  * 仪表盘 Dashboard
  * 文章管理
  * 评论审核
  * 友链管理
* 每个导航项应包含图标 + 文本
* 当前选中项：

  * 背景为极浅主题色
  * 左侧有 `4px` 主题色指示条
  * 文字高亮为主题色
* 非选中项 hover 时：

  * 背景轻微发亮
  * 文本颜色略提升
* 小屏幕下 Sidebar 不应永久占据左侧，应退化为抽屉菜单

### 右侧主内容区

* 顶部为 macOS 风格控制栏

  * 包含红黄绿三色占位按钮或拟态窗口控制点
  * 标题区可显示当前页面名
  * 可预留搜索框 / 用户信息入口位置
* 下方为仪表盘内容区，包含欢迎信息、统计卡片、快捷入口、最近动态等板块

### 数据卡片 (Stats Cards)

* 使用 CSS Grid 排列
* 建议桌面端 3~4 列，自适应换行
* 每张卡片必须是独立的 `LiquidGlassCard.vue`
* 内容包含：

  * 指标名称
  * 大号数字
  * 环比/趋势小标签（可选）
  * 小图标（可选）
* 数字字体必须为 `JetBrains Mono` 或 `Fira Code`
* 数字、标题、辅助说明层级要清晰，不可挤在一起像财务报表失控现场

### 可扩展区块建议

可预留以下模块容器，即使暂时使用假数据，也要把布局先定下来：

* 最近文章
* 最新评论待审核
* 系统状态
* 快捷操作入口（新建文章、查看站点、管理友链）

### 状态与空态

* 首次加载时显示 Skeleton
* 数据为空时显示占位文案与简洁图标
* 请求失败时展示错误反馈区块，而不是让页面留下一片哲学性的空白

### 权限与守卫

* `/admin` 必须视为受保护路由
* 无登录态时应跳转至 `/login`
* 页面初始化时可显示短暂壳层 Loading，避免未鉴权内容闪烁

## 3. 留言板 (Guestbook)

路径：`src/pages/guestbook.astro`

### 页面定位

留言板不是普通评论区，而是一个带有温度感的访客交流空间。视觉上应更偏向 Apple Messages 与轻量瀑布流展示之间的融合：柔和、整洁、可阅读、有互动温度。

### 页面结构

* 顶部为页面标题与说明文案
* 中间为留言输入区
* 下方为留言卡片流列表
* 页面整体内容宽度建议限制在舒适阅读范围内，如 `max-w-6xl`

### 顶部输入区

* 使用宽敞的 `LiquidGlassCard.vue`
* 内部包含：

  * 昵称输入框
  * 网址输入框（可选）
  * 多行留言输入框
  * 发送按钮
* 发送按钮位于文本框区域右下或底部右对齐
* 按钮 hover 时 `scale-105`
* 提交中按钮需禁用并显示 loading 动画

### 输入体验要求

* 文本框支持合理最小高度
* 输入内容较长时允许自然扩展或滚动，但初始状态不能太矮
* 占位文案要友好、有温度，但不要故作可爱过头
* 校验失败需有温和提示，不可直接清空用户内容

### 留言列表展示

* 优先采用 Masonry（瀑布流）或单列宽卡片流
* 在移动端必须自然退化为单列
* 每条留言卡片内部结构：

  * 左侧头像
  * 右侧内容区

    * 昵称 / ID
    * 时间戳（灰色小字）
    * 留言正文
    * 若填写网址，可作为低调入口展示
* 留言正文必须支持换行与长文本折行，不能溢出卡片

### 卡片视觉细节

* 卡片必须为 `LiquidGlassCard.vue`
* 卡片之间留白要充足，避免瀑布流显得拥挤
* 可为不同留言卡片引入极轻微高度差，营造自然感
* 头像建议固定尺寸，避免因用户数据不一致导致排版散架

### 交互动画

* 新留言提交成功后：

  * 新卡片插入列表顶部或合理位置
  * 通过 Vue `<TransitionGroup>` 或 Tailwind 自定义 `@keyframes` 实现
  * 动画效果为“从下方滑入 + 淡入”
* Hover 卡片时可有极轻微阴影增强
* 不允许出现弹飞、抖动、夸张缩放等廉价动画

### 列表状态

必须实现以下状态区分：

* 加载中：显示骨架卡片
* 无留言：显示空态说明
* 加载失败：显示错误提示与重试入口
* 提交成功：局部插入动画 + 成功提示
* 提交失败：保留已输入内容并提示重试

### 额外建议

* 可预留“只看站长回复”或“按时间排序”筛选位，但若未实现功能，不要生成假交互按钮
* 若后端未来接入防刷机制，前端按钮禁用与冷却态要预留兼容空间

## 4. 友情链接 (Friends Links)

路径：`src/pages/friends.astro`

### 页面定位

友情链接页不仅是资源列表，更是站点宇宙的一块展示面板。风格上应参考 macOS Launchpad 式的整洁网格，突出秩序、呼吸感与卡片触感。

### 页面结构

* 顶部说明区
* 操作按钮区
* 友链网格列表区

### 顶部说明区

* 使用优雅、克制的文案介绍交换规则
* 可包含：

  * 本站介绍
  * 交换要求
  * 申请方式
* 说明文案建议分段排布，不宜一大坨
* 下方可放置“申请交换”按钮

  * 主题色边框
  * hover 时背景轻微染上主题色
  * 可预留后续弹窗表单接入

### 友链网格 (Grid)

* 必须使用响应式网格：

  * `grid-cols-1`
  * `md:grid-cols-2`
  * `lg:grid-cols-3`
* 卡片之间的 gap 要足够大，保证 Launchpad 式呼吸感
* 整体宽度不宜无限拉宽

### 友链卡片细节

* 强制使用 `LiquidGlassCard.vue`
* Flex 布局：

  * 左侧为 Avatar / Logo
  * 右侧为文本信息
* Avatar：

  * 支持圆形或圆角方形
  * `object-cover`
  * 尺寸统一
* 文本区：

  * 网站名称：粗体
  * 简介：灰色辅助文字
  * 超长简介必须 `truncate` 或限制行数
* 可补充低调展示域名，但不可抢主标题风头

### 交互动效

* Hover 时必须有：

  * `scale-105`
  * 阴影从 `shadow-lg` 过渡到 `shadow-xl`
  * 边框或背景透明层微妙增强
* 点击后应具备明确可点击反馈
* 整体触感应柔和高级，而不是像电商商品卡狂跳

### 状态规范

* 加载中：显示若干占位卡片
* 空数据：显示空态说明
* 请求失败：显示错误提示与重试按钮
* 外链打开建议使用新标签页，并附加安全属性

# 六、交互实现边界与工程要求

## 1. Astro 与 Vue Islands 的职责边界

* Astro 负责页面骨架、SEO 基础结构、静态容器与首屏结构输出
* Vue Islands 负责：

  * 表单输入
  * 本地响应式状态
  * 异步请求
  * Loading / Error / Success 状态
  * 动画插入与切换
* 不允许把整页全部粗暴塞进一个大 Vue 组件里失去 Astro 的意义

## 2. 数据请求边界

* 当前阶段若后端接口未全部完成，可使用 mock 数据或占位结构
* 但组件的 props、状态流、空态与错误态必须按真实接口思路设计
* 禁止只做“静态摆拍页面”而不考虑后续接 API 的可扩展性

## 3. 代码组织边界

* 页面文件只负责拼装，不应塞满复杂业务逻辑
* 复杂交互、列表项、表单块必须拆组件
* 通用动画 class、状态样式、图标逻辑应复用，不可重复制造样式碎片

# 七、AI 编码执行指令（增强版）

1. 你的职责是 **Vue 3 + Astro 前端架构师**，不是单纯页面拼接工具。输出代码前，必须先分析页面结构、组件复用关系与状态流。
2. 生成每个页面前，必须优先阅读并理解 `LiquidGlassCard.vue` 的设计意图，确保视觉语言统一。
3. 所有表单类交互（登录、留言提交）必须通过前端按需加载的 Vue Islands 实现，不允许在 Astro 页面内硬写整套响应式逻辑。
4. 对四个页面，必须先分析它们之间的共同组件与共享状态逻辑，再开始逐页编码。
5. 每生成一个页面时，输出顺序必须为：

   * 页面目标分析
   * 组件拆分建议
   * 页面代码
   * 依赖的子组件代码
   * 相关 Nano Stores 代码（若本页需要）
   * 接入说明
6. 若某页依赖的公共组件尚不存在，必须优先补齐公共组件，而不是在页面内部重复造轮子。
7. 严格禁止使用 GORM、Gin 等后端框架语法；前端层严禁脱离 Tailwind 类名直接写内联样式。
8. 若用户尚未确认开始编码，必须先询问：

**“是否开始生成登录页面？”**
